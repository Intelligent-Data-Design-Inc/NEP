# V1.10.0 Sprint 4: Add example: fill_values.c — Fill Value Performance

**Objective**: Create `examples/performance/fill_values.c` demonstrating fill value handling performance across classic (netCDF-3) and netCDF-4/HDF5 formats. The example shows how fill values affect write/read performance and file size, with CSV output suitable for plotting.

---

## Background

NetCDF supports fill values — a special value indicating missing or unwritten data. When fill mode is enabled (default), NetCDF automatically writes fill values to any data not explicitly written. This sprint creates a performance example that:

- Compares fill mode ON vs OFF write performance
- Shows performance difference between classic and netCDF-4 formats
- Measures the overhead of fill value checking during reads
- Demonstrates `_FillValue` attribute handling

**Dataset**: Same 3D temperature, 500×180×360, used in sprints 1–3 for consistency.

### 1. Create `examples/performance/fill_values.c`

**File**: `examples/performance/fill_values.c`

**Program structure**:

```c
/* Header: format,fill_mode,write_s,read_s,file_bytes */

static int run_test(const char *path, int format, int fill_mode) { ... }

int main(void) {
    printf("format,fill_mode,write_s,read_s,file_bytes\n");
    
    /* Classic format tests */
    if (run_test("fill_classic_fillon.nc", NC_CLOBBER, 1)) return 1;
    if (run_test("fill_classic_filloff.nc", NC_CLOBBER, 0)) return 1;
    
    /* NetCDF-4/HDF5 tests */
    if (run_test("fill_nc4_fillon.nc", NC_NETCDF4 | NC_CLOBBER, 1)) return 1;
    if (run_test("fill_nc4_filloff.nc", NC_NETCDF4 | NC_CLOBBER, 0)) return 1;
    
    /* Cleanup */
    remove("fill_classic_fillon.nc");
    remove("fill_classic_filloff.nc");
    remove("fill_nc4_fillon.nc");
    remove("fill_nc4_filloff.nc");
    
    return 0;
}
```

**`run_test(path, format, fill_mode)` logic**:
1. Create file with specified format using `nc_create()`
2. Set fill mode via `nc_set_fill(ncid, fill_mode ? NC_FILL : NC_NOFILL, &old_mode)`
3. Define 3D dimensions (500×180×360) matching sprints 1–3
4. Define NC_FLOAT `temperature` variable
5. Optionally set `_FillValue` attribute (when fill_mode=1)
6. Write pass: time `nc_enddef()` followed by `nc_put_var_float()` with synthetic data
7. Close and stat the file to get on-disk size
8. Read pass: time `nc_open()` + `nc_get_var_float()` + `nc_close()`
9. Print one CSV row: `format_name,fill_mode,write_s,read_s,file_bytes`

**Dataset constants** (match sprints 1–3):
```c
#define NZ  500    /* time */
#define NY  180    /* lat */
#define NX  360    /* lon */

static void generate_data(float *data)
{
    size_t t, y, x, idx = 0;
    for (t = 0; t < NZ; t++)
        for (y = 0; y < NY; y++)
            for (x = 0; x < NX; x++)
                data[idx++] = 280.0f + t * 0.1f + y * 0.01f + x * 0.001f;
}
```

**Error handling**: Use standard NEP `NC_CHK` pattern:
```c
#define NC_CHK(err) do { \
    int ret = (err); \
    if (ret != NC_NOERR) { \
        fprintf(stderr, "Error at %s:%d: %s\n", __FILE__, __LINE__, nc_strerror(ret)); \
        return ret; \
    } \
} while(0)
```

**Doxygen file header**: Explain fill value concept, NC_FILL vs NC_NOFILL modes, and how fill value handling affects I/O performance.

## Notes

- **Fill mode behavior**: With `NC_FILL`, NetCDF writes fill values to all data before user writes. This adds overhead at write time but ensures no unwritten data remains. With `NC_NOFILL`, writes are faster but unwritten data locations contain undefined values.
- **Classic vs netCDF-4 fill handling**: Classic files handle fill values at the library level; netCDF-4 files can use per-variable fill values via `nc_def_var_fill()`.
- **No pass/fail assertions**: output is informational; CI only checks exit code 0.
- **`ENABLE_BENCHMARKS` defaults OFF**: protects CI from the 4× 129 MB writes; developers enable locally.
- **`stat()` availability**: POSIX; available on Linux and macOS. No Windows support needed.
