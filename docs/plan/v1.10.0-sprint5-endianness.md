# v1.10.0 Sprint 5: Add example: endianness.c — Endianness Performance

---

## Objective

Create `endianness.c`, a C performance example that demonstrates byte order (endianness) handling in NetCDF-4/HDF5 files. The example measures and compares write/read performance across native, little-endian, and big-endian byte orders.

---

## Background

NetCDF-4, built on HDF5, supports explicit control over the byte order used to store variable data. By default, data is stored in the platform's native byte order (NC_ENDIAN_NATIVE). However, users can specify NC_ENDIAN_LITTLE or NC_ENDIAN_BIG to ensure cross-platform compatibility or match requirements of downstream tools.

When a non-native byte order is specified, the HDF5 library must perform byte-swapping during I/O operations. This sprint creates a performance example to quantify the overhead of explicit byte order conversion compared to native byte order storage.

## Notes

### Byte Order Considerations

On little-endian platforms (x86, x86_64, ARM), NC_ENDIAN_NATIVE is equivalent to NC_ENDIAN_LITTLE. Setting NC_ENDIAN_BIG on these platforms will require explicit byte-swapping during I/O.

On big-endian platforms (some PowerPC, IBM Z), NC_ENDIAN_NATIVE is equivalent to NC_ENDIAN_BIG. Setting NC_ENDIAN_LITTLE on these platforms will require explicit byte-swapping.

The performance example will demonstrate this platform-dependent behavior. On typical x86_64 Linux systems:
- NC_ENDIAN_NATIVE and NC_ENDIAN_LITTLE should have similar performance
- NC_ENDIAN_BIG may show overhead due to byte-swapping

### Dataset Consistency

All v1.10.0 performance examples (sprints 1–5) use the same dataset dimensions:
- 500×180×360 NC_FLOAT temperature array
- ~129 MB uncompressed
- This ensures results are comparable across examples
