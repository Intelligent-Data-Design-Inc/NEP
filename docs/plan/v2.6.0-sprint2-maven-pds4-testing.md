# v2.6.0 Sprint 2: PDS4 Testing with Maven Data

## Sprint Goal

Validate the NEP PDS4 reader against real NASA MAVEN NGIMS mission data. Both Maven
files already exist in `test/data/PDS4/`. The sprint adds two new mission test
functions to `test/tst_pds4_udf.c`, fixes any type-mapping gaps exposed by the Maven
labels, and wires the new data files into both build systems.

---

## Scope

- Fix `ASCII_NonNegative_Integer` and `ASCII_Date_Time` type mappings in `src/pds4file.c`
- Add `test_mission_maven_l1b()` to `test/tst_pds4_udf.c`
- Add `test_mission_maven_l3()` to `test/tst_pds4_udf.c`
- Wire Maven file pairs into `test/CMakeLists.txt` and `test/Makefile.am`
- Document any new known limitations discovered during testing

**Out of scope:** New PDS4 features, write support, performance optimization.

## Major Tasks (dependency order)

### 1. Fix type-mapping gaps in `src/pds4file.c`

The Maven L3 label uses two PDS4 types not currently in the type table:

- `ASCII_NonNegative_Integer` → `NC_INT64`
- `ASCII_Date_Time` → `NC_STRING`

Locate the type-mapping table in `src/pds4file.c` and add these two entries.
Build and confirm the L3 label opens without error before writing tests.

### 2. Empirically determine field-name normalization for hyphens

The L3 label contains the field name `EXO-ALT` (hyphen). The existing reader
normalizes spaces to underscores (confirmed by `test_mission_lcs_9p()` using
`Spec_Num`). Build and attempt `nc_inq_varid(..., "EXO-ALT", ...)` vs
`nc_inq_varid(..., "EXO_ALT", ...)` to determine the actual behaviour, then
hard-code the correct name in the test.

If hyphens are not normalized, consider whether to normalize them in `pds4file.c`
(consistent with spaces) or document as-is. Decision to be made during implementation.

### 3. Wire Maven file pairs into build systems

**CMake (`test/CMakeLists.txt`):**
Add both XML + CSV pairs to the `file(COPY ...)` or `configure_file()` rules that
copy PDS4 test data to the build directory. Follow the existing pattern for
`cassini_hrd/`, `messenger_tnmap/`, and `lcs_9p/` entries.

**Autotools (`test/Makefile.am`):**
Add both XML + CSV pairs to `dist_check_DATA` (or equivalent) so they are
distributed and available during `make check`. Follow the existing pattern.

### 4. Add `test_mission_maven_l1b()` to `test/tst_pds4_udf.c`

**Pattern:** mirror `test_mission_cassini_hrd()` / `test_mission_lcs_9p()`.

**Assertions:**
- `nc_open("data/PDS4/mvn_ngi_l1b_cal-hk-058943_20250101T023235_v01_r02.xml", NC_NOWRITE, &ncid)` succeeds
- Child group `"mvn_ngi_l1b_cal-hk-058943_20250101T023235_v01_r02.csv"` exists
- `nvars == 324` (one variable per field)
- `record` dimension has length `26121`
- `TIME` variable is `NC_DOUBLE`
- Light data read: `nc_get_vara_double(..., "TIME", start={0}, count={2}, buf)` succeeds and `buf[0] > 0.0`

**Define macro:** `#define PDS4_MAVEN_L1B_FILE "data/PDS4/mvn_ngi_l1b_cal-hk-058943_20250101T023235_v01_r02.xml"`

### 5. Add `test_mission_maven_l3()` to `test/tst_pds4_udf.c`

**Pattern:** mirror `test_mission_lcs_9p()`.

**Assertions:**
- `nc_open("data/PDS4/.../mvn_ngi_l3_res-sht-58942_20250101T010116_v06_r03.xml", NC_NOWRITE, &ncid)` succeeds
- Child group `"mvn_ngi_l3_res-sht-58942_20250101T010116_v06_r03.csv"` exists
- `nvars == 15`, `record` dimension has length `2`
- `T_UTC` variable is `NC_STRING`
- `T_UNIX` variable is `NC_DOUBLE`
- `TEMPERATURE` variable is `NC_DOUBLE`
- Data read: `T_UNIX[0]` ≈ `1735698148.655328` (within 0.001)
- Data read: `SCALE_HEIGHT[0]` ≈ `3.210464` (within 0.001)
- Data read: `TEMPERATURE[0]` ≈ `51.332605` (within 0.01)

**Note on `EXO-ALT`:** verify name at build time; add assertion once normalization
behaviour is confirmed.

**Define macro:** `#define PDS4_MAVEN_L3_FILE "data/PDS4/mvn_ngi_l3_res-sht-58942_20250101T010116_v06_r03.xml"`

### 6. Call new test functions from `main()` in `test/tst_pds4_udf.c`

Add after the existing `test_mission_lcs_9p()` call:

Update the final success message to reflect the new tests.
