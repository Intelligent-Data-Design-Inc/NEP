# v1.10.0 Sprint 7: Add example: szip.c — SZIP Compression Performance

---

## Objective

Create `szip.c`, a C performance example that demonstrates how SZIP compression
parameters (`options_mask` and `pixels_per_block`) affect compression ratio and
I/O throughput on a realistic scientific dataset. Output is CSV suitable for
plotting. Add `plot_szip.py` to visualise results. The program is gated by
`ENABLE_BENCHMARKS` and does not run in regular CI.

Requires NetCDF-C built with SZIP write support. The local szip-enabled build is
at `/usr/local/netcdf-c-4.10.0_szip`, backed by HDF5 at
`/usr/local/hdf5-2.1.0_szip` and the szip library at `/usr/local/szip-2.1.1`.

---

## Background

SZIP (Lossless Scientific Data Compression) is a lossless compression algorithm
developed at JPL for compressing scientific data. It is integrated into HDF5 as
filter ID 4. NetCDF-4 exposes it via `nc_def_var_szip()`.

Two coding methods are available:

- `NC_SZIP_NN` (options_mask = 32): Nearest-neighbor coding. Generally
  achieves better ratios on smoothly varying floating-point data.
- `NC_SZIP_EC` (options_mask = 4): Entropy coding. Often faster but lower
  ratio on continuous fields.

The `pixels_per_block` parameter (even values 2 to 32) controls the block size
used by the SZIP encoder. Smaller values process more blocks with lower overhead
per block; larger values (especially 32) often improve ratio by giving the
encoder more context, at some increase in CPU cost.

Unlike deflate and zstandard, SZIP does not have a separate shuffle filter
interaction. The shuffle filter can still be applied via `nc_def_var_deflate()`
before `nc_def_var_szip()` for floating-point data, but SZIP already operates on
individual sample values and its own internal byte ordering. This sprint does not
combine SZIP with shuffle; it sweeps the two coding methods and all valid
`pixels_per_block` values to characterise the parameter space.

## Program Structure

### Constants

```c
#define NZ 500
#define NY 180
#define NX 360
#define TMP_FILE "szip_tmp.nc"
#define CHUNK_Z 10
#define CHUNK_Y 45
#define CHUNK_X 90
#define NVALS ((size_t)NZ * NY * NX)
#define UNCOMPRESSED_BYTES ((double)NVALS * sizeof(float))
#define ERRCODE 2
#define ERR(e) {printf("Error: %s\n", nc_strerror(e)); exit(ERRCODE);}
```

### Coding methods array

```c
static const int PPB_VALUES[] = {2, 6, 10, 18, 30};  /* even divisors of CHUNK_X=90 */
#define N_PPB 5
```

### `run_one(data, options_mask, pixels_per_block)` logic

1. `nc_create(TMP_FILE, NC_NETCDF4 | NC_CLOBBER, &ncid)`
2. Define dims `time`, `lat`, `lon`
3. `nc_def_var(ncid, "temperature", NC_FLOAT, 3, dimids, &varid)`
4. `nc_def_var_chunking(ncid, varid, NC_CHUNKED, chunksizes)`
5. `nc_def_var_szip(ncid, varid, options_mask, pixels_per_block)`
6. `nc_enddef(ncid)`
7. `nc_inq_var_szip(ncid, varid, &mask_out, &ppb_out)` — verify settings
8. `nc_put_var_float(ncid, varid, data)`
9. `nc_close(ncid)` — record `write_s`
10. `stat(TMP_FILE, &st)` — `compressed_bytes`, compute `ratio`
11. `nc_open(TMP_FILE, NC_NOWRITE, &ncid)`
12. `nc_get_var_float(ncid, varid, data)` — record `read_s`
13. `nc_close(ncid)`
14. `remove(TMP_FILE)`
15. Print CSV row

### CSV output

```
pixels_per_block,compressed_bytes,ratio,write_s,read_s
```

### `main()` loop

```c
for (pi = 0; pi < N_PPB; pi++)
    run_one(data, PPB_VALUES[pi]);
```

## Plotting: `plot_szip.py`

Reads `szip_results.csv`, writes `szip_performance.jpg`.

Three subplots (matching `plot_zstandard.py` style):

1. **Compression ratio** vs `pixels_per_block`, one line per coding method
2. **Write time (s)** vs `pixels_per_block`, one line per coding method
3. **Read time (s)** vs `pixels_per_block` (same y-axis scale as write time),
   one line per coding method

Line styles: solid/filled for NN, dashed/open for EC (black and white).
X-axis ticks at each `pixels_per_block` value.
