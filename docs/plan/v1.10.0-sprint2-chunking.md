# V1.10.0 Sprint 2: Add example: chunking.c — Chunk Size Performance

**Objective**: Create `examples/performance/chunking.c` demonstrating how chunk shape affects I/O performance across two contrasting access patterns (optimal vs. worst-case), outputting CSV suitable for plotting. The program is guarded by `ENABLE_BENCHMARKS` and does not run in regular CI.

---

## Background

Chunk size selection is one of the most impactful and least-understood performance knobs in NetCDF-4. The right chunk shape depends entirely on how the data will be accessed. A chunk layout tuned for time-series access (one time step at a time) is catastrophic for spatial access (one lat/lon slice at a time), and vice versa.

Sprint 1 (`cache_tuning.c`) demonstrated how to configure the cache once a chunk shape is chosen. Sprint 2 shows how to choose the shape in the first place by measuring read time across multiple shapes and access patterns.

**Dataset**: Same 3D temperature variable as sprint 1 — 500×180×360 (time×lat×lon) — so both examples work on a consistent, meteorologically representative domain and can share the test file if desired.

**Access patterns** (2 total):
1. **Time slab** (`start=[t,0,0]`, `count=[1,180,360]`): reads one complete horizontal field at a time; typical forecast model output access
2. **Column profile** (`start=[0,lat,lon]`, `count=[500,1,1]`): reads a full time series at a single grid point; typical station extraction

**Chunk shapes** (2 total):
1. **Time-optimized** (`1×180×360`): one complete horizontal field per chunk; ideal for time-slab access, worst-case for column access
2. **Column-optimized** (`500×1×1`): one complete time series per grid point per chunk; ideal for column access, worst-case for time-slab access

This deliberate 2×2 contrast makes the performance difference dramatic and immediately instructive.

## Tasks

### 1. Create `examples/performance/chunking.c`

**File**: `examples/performance/chunking.c`

The program:
1. Creates (or recreates) the 500×180×360 temperature file **twice** — once with time-optimized chunks, once with column-optimized chunks
2. For each file, runs both access patterns (time slab and column profile), timing each with `gettimeofday()`
3. Prints one CSV row per measurement:  `chunk_shape,access_pattern,elapsed_s,MB_per_s`
4. Prints a header row first, then 4 data rows (2 files × 2 patterns)
5. Exits 0

**Key constants**:
```c
#define NZ  500    /* time */
#define NY  180    /* lat */
#define NX  360    /* lon */

/* Chunk shape A: time-optimized (one full horizontal field) */
#define CHUNK_Z_TIME  1
#define CHUNK_Y_TIME  180
#define CHUNK_X_TIME  360

/* Chunk shape B: column-optimized (full time series at one point) */
#define CHUNK_Z_COL   500
#define CHUNK_Y_COL   1
#define CHUNK_X_COL   1
```

**CSV output format**:
```
chunk_shape,access_pattern,elapsed_s,MB_per_s
time_optimized,time_slab,0.123,2341.5
time_optimized,column_profile,4.567,63.0
column_optimized,time_slab,3.890,73.9
column_optimized,column_profile,0.089,3202.6
```

**Data volume per test**:
- Time slab: reads 500 slabs × (180×360×4 bytes) = ~129 MB total
- Column profile: reads 180×360 = 64,800 points × (500×4 bytes) ≈ 129 MB total (same volume, different access)

**Timing**: Use `gettimeofday()` wrapping the full loop of reads; report total elapsed and MB/s.

**Program structure**:
```c
/* write_file(path, chunk_z, chunk_y, chunk_x) — create file with given chunking */
/* time_access_time_slab(path, label) — read 500 time slabs, print CSV row */
/* time_access_column(path, label) — read 64800 column profiles, print CSV row */

int main() {
    printf("chunk_shape,access_pattern,elapsed_s,MB_per_s\n");

    write_file("chunking_time.nc", CHUNK_Z_TIME, CHUNK_Y_TIME, CHUNK_X_TIME);
    time_access_time_slab("chunking_time.nc", "time_optimized");
    time_access_column("chunking_time.nc", "time_optimized");

    write_file("chunking_col.nc", CHUNK_Z_COL, CHUNK_Y_COL, CHUNK_X_COL);
    time_access_time_slab("chunking_col.nc", "column_optimized");
    time_access_column("chunking_col.nc", "column_optimized");

    remove("chunking_time.nc");
    remove("chunking_col.nc");
    return 0;
}
```

**Error handling**: Use `NC_CHK` macro (same pattern as `cache_tuning.c`).

**Doxygen header**: File-level doc explaining chunk shape selection, meteorological motivation for the two patterns, and how to plot the CSV output.

### 3. Update `examples/performance/CMakeLists.txt`

Add `chunking` executable alongside `cache_tuning`. Gate the test registration on `ENABLE_BENCHMARKS` (both programs):

```cmake
# Chunking example
add_executable(chunking chunking.c)
target_compile_options(chunking PRIVATE ${NC_CFLAGS_LIST})
target_link_directories(chunking PRIVATE ${HDF5_LIB_DIR})
target_link_libraries(chunking PRIVATE ${NC_LIBS_LIST} hdf5 hdf5_hl m)

# Performance examples only run as tests when ENABLE_BENCHMARKS=ON
if(ENABLE_BENCHMARKS)
    target_compile_definitions(cache_tuning PRIVATE ENABLE_BENCHMARKS=1)
    target_compile_definitions(chunking PRIVATE ENABLE_BENCHMARKS=1)
    if(BUILD_TESTING)
        add_test(NAME cache_tuning COMMAND cache_tuning
                 WORKING_DIRECTORY ${CMAKE_CURRENT_BINARY_DIR})
        add_test(NAME chunking COMMAND chunking
                 WORKING_DIRECTORY ${CMAKE_CURRENT_BINARY_DIR})
    endif()
endif()
```

Remove the unconditional `add_test(NAME cache_tuning ...)` block from sprint 1 (replace with the gated version above).

## Notes

- **Writing the data**: `write_file()` writes synthetic data (e.g., `temp[t][y][x] = 280.0f + t * 0.1f`) — correctness is not the point, disk I/O patterns are
- **Column profile loop**: 180×360 = 64,800 reads of a `[500,1,1]` hyperslab is slow with time-optimized chunks — this is intentional and makes the contrast vivid
- **No pass/fail timing assertions**: output is informational; CI only checks exit code 0
- **`ENABLE_BENCHMARKS` defaults OFF**: protects CI from slow disk benchmarks; developers enable locally for profiling work
- **Same dataset size as sprint 1**: consistent 500×180×360 domain simplifies comparison across both examples
