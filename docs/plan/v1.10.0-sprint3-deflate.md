# V1.10.0 Sprint 3: Add example: deflate.c — Deflate Compression Performance

**Objective**: Create `examples/performance/deflate.c` demonstrating how zlib deflate levels (0–9) and the shuffle filter affect compression ratio and I/O throughput. Output is CSV suitable for plotting. Add `plot_deflate.py` to visualise results. The program is gated by `ENABLE_BENCHMARKS` and does not run in regular CI.

---

## Background

NetCDF-4/HDF5 supports per-variable deflate compression via `nc_def_var_deflate()`. The `level` parameter (0 = no compression, 9 = maximum) trades CPU time for smaller files. The shuffle filter (`shuffle=1`) reorders bytes before compression to improve ratios for floating-point data; it is almost always beneficial and costs little.

Sprint 1 (`cache_tuning.c`) showed how to configure the HDF5 chunk cache. Sprint 2 (`chunking.c`) showed how chunk *shape* determines which access patterns are fast. Sprint 3 completes the trilogy by showing how *compression settings* affect both file size and I/O throughput.

---

## The deflate.c Program

`examples/performance/deflate.c` is a self-contained C benchmark that measures the effect of NetCDF-4 deflate compression settings on file size and I/O throughput. It creates a 500×180×360 NC_FLOAT temperature variable — the same meteorologically realistic domain used by `cache_tuning.c` and `chunking.c` — with chunk shape 10×45×90, then iterates all 20 combinations of deflate level (0–9) and shuffle filter (off/on). For each combination it times the full write (`nc_put_var_float`), stats the file to get the on-disk size, and times the full read (`nc_get_var_float`). Results are printed as CSV to stdout with columns `deflate_level,shuffle,compressed_bytes,ratio,write_s,read_s`.

The program calls `nc_def_var_deflate(ncid, varid, shuffle, 1, level)` to set compression and `nc_inq_var_deflate()` immediately after `nc_enddef()` to verify that NetCDF-C accepted the settings before any data is written. The temporary file (`deflate_tmp.nc`) is created with `NC_CLOBBER` and removed with `remove()` after each iteration, so no disk space accumulates across the 20 runs. The uncompressed dataset is ~129 MB; at the extremes (level 9, shuffle=on) the on-disk file is 2.5 MB, so the total I/O across all 20 iterations is manageable on any development machine.

To run the benchmark and generate the plot:

```bash
./deflate 2>/dev/null > deflate_results.csv
python3 plot_deflate.py   # writes deflate_performance.jpg
```

The resulting chart (`deflate_performance.jpg`) has three subplots: compression ratio, write time, and read time — each plotted against deflate level with separate lines for shuffle=off (solid) and shuffle=on (dashed). The data in `deflate_results.csv` can also be imported directly into any spreadsheet or analysis tool.

## Benchmark Results and Analysis

Results below are from a 500×180×360 NC_FLOAT temperature variable (~129 MB uncompressed, chunk shape 10×45×90) on a local Linux workstation. See `deflate_results.csv` and `deflate_performance.jpg`.

### The shuffle filter is the single most important setting

The shuffle filter reorders bytes by significance before compression: all the most-significant bytes of adjacent floats are grouped together, all the least-significant bytes together, and so on. For smoothly varying scientific data — temperatures, pressures, wind speeds — adjacent values are numerically similar, so their high-order bytes are nearly identical. Grouping them gives the deflate algorithm long runs of nearly-constant bytes, which it compresses extremely well.

The data bear this out dramatically. At deflate level 1, the shuffle filter reduces the file from 16.2 MB to 4.4 MB — a further 3.7× size reduction at no meaningful cost. Across all levels, shuffle=on consistently achieves 3–4× better ratios than shuffle=off. At level 9, shuffle=off reaches a ratio of 13.4×; shuffle=on reaches 52.6×. The uncompressed file is 129 MB; with shuffle and level 9 it shrinks to 2.5 MB. **Always enable the shuffle filter for floating-point data.**

### Use deflate level 1: the write-time cliff starts here

The compression ratio curve is steep between level 0 and level 1, then flattens quickly. Without shuffle, level 1 achieves a ratio of 7.6× (write time 0.75 s); level 9 achieves 13.4× (write time 2.79 s) — less than 2× more compression for nearly 4× more CPU time. With shuffle, level 1 achieves 28.3× (write time 0.64 s); level 9 achieves 52.6× (write time 1.17 s) — again less than 2× more compression for almost 2× more CPU time. Meanwhile, files written at level 1 with shuffle are already small enough that read time is well below that of the uncompressed file (0.20 s vs 0.06 s uncompressed — slightly slower to read due to decompression CPU cost, but the smaller file means less I/O). **For most scientific workflows, deflate level 1 with shuffle gives the best balance of compression ratio, write throughput, and read performance.**

### Read time is largely insensitive to compression level

Unlike write time, read time barely changes across levels 1–9: it ranges from 0.19–0.27 s with shuffle=on and 0.19–0.27 s with shuffle=off. The decompression algorithm does not vary in cost the way the compression algorithm does — zlib decompression is O(output size) regardless of the level used during compression. In practice, higher compression levels make files read *slightly* faster (less data to transfer from disk) but cost substantially more at write time. The crossover point depends on the read/write ratio of the workflow; for most data that is written once and read many times, even level 6 or 7 may be justified, but level 1 is the safe default.

**Dataset**: Same 500×180×360 NC_FLOAT temperature as sprints 1 and 2 — a meteorologically realistic domain providing a consistent comparison basis across all three examples.

**Measurements per row**:
- deflate level (0–9)
- shuffle on/off
- compressed file size in bytes
- compression ratio (uncompressed / compressed)
- write time in seconds
- read time (full variable, single `nc_get_var_float`) in seconds

This produces 20 CSV rows (10 levels × 2 shuffle states), giving clear curves of ratio and throughput vs. level for both shuffle settings.

## Tasks

### 1. Create `examples/performance/deflate.c`

**File**: `examples/performance/deflate.c`

**Dataset constants** (match sprint 1):
```c
#define NZ  500    /* time */
#define NY  180    /* lat */
#define NX  360    /* lon */
```

**Program structure**:

```c
/* Header: deflate_level,shuffle,compressed_bytes,ratio,write_s,read_s */

static int run_one(int level, int shuffle) { ... }

int main(void) {
    int level, shuffle;
    printf("deflate_level,shuffle,compressed_bytes,ratio,write_s,read_s\n");
    for (shuffle = 0; shuffle <= 1; shuffle++)
        for (level = 0; level <= 9; level++)
            if (run_one(level, shuffle)) return 1;
    return 0;
}
```

**`run_one(level, shuffle)` logic**:
1. Generate synthetic NC_FLOAT data (`280.0f + t * 0.1f + y * 0.01f + x * 0.001f`) into a heap-allocated `NZ×NY×NX` buffer.
2. Write pass: time `nc_create → nc_def_var → nc_def_var_deflate(level, shuffle) → nc_put_var_float → nc_close` with `gettimeofday()` wrapping just the write. File name: `deflate_tmp.nc` (overwritten each iteration with `NC_CLOBBER`).
3. Stat the file with `stat()` (POSIX) to get compressed bytes.
4. Compute ratio: `(NZ * NY * NX * sizeof(float)) / compressed_bytes`.
5. Read pass: time `nc_open → nc_get_var_float → nc_close`.
6. Print one CSV row.
7. `remove("deflate_tmp.nc")`.

**Error handling**: Use the `NC_CHK` macro (same pattern as `cache_tuning.c` and `chunking.c`).

**Doxygen file header**: Explain deflate levels, shuffle filter benefit for floats, and how to interpret the CSV output. Document `nc_def_var_deflate()` parameters.

**Key API functions demonstrated**:
| Function | Purpose |
|---|---|
| `nc_def_var_deflate()` | Set deflate level and shuffle filter on a variable |
| `nc_inq_var_deflate()` | Query deflate and shuffle settings on an open variable |
| `nc_put_var_float()` | Write entire variable in one call |
| `nc_get_var_float()` | Read entire variable in one call |
| `stat()` | Obtain compressed file size on disk |
| `gettimeofday()` | Wall-clock timing |

## Notes

- **Chunking for deflate**: `nc_def_var_deflate()` requires the variable to be chunked. Use `nc_def_var_chunking()` with a single chunk covering the full time dimension but HDF5-default sizes for lat/lon, or use a fixed chunk shape like `10×45×90` (matching sprint 1) to make compression behaviour reproducible. The plan doc notes this explicitly so the implementer knows to add the `nc_def_var_chunking()` call.
- **Level 0**: `nc_def_var_deflate(ncid, varid, shuffle, 1, 0)` enables deflate at level 0 (store with no compression effort); ratio ≈ 1.0 for float data with little spatial structure; useful as a baseline.
- **No pass/fail assertions**: output is informational; CI only checks exit code 0.
- **`ENABLE_BENCHMARKS` defaults OFF**: protects CI from the ~20× 129 MB writes; developers enable locally.
- **`stat()` availability**: POSIX; available on Linux and macOS. No Windows support needed.
- **`nc_inq_var_deflate()` verification**: calling it immediately after `nc_enddef()` (before `nc_put_var_float()`) verifies that the deflate settings were accepted without an extra file open/close cycle.
