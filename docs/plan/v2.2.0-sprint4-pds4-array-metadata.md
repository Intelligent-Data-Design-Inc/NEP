# V2.2.0 Sprint 4: Read PDS4 Array Metadata

**Objective**: Convert the PDS4 XML label metadata for array products into the netCDF-4 in-memory model so that `nc_inq_*` functions return correct groups, dimensions, variables, and attributes. No data is read yet.

---

## Background

A PDS4 `Product_Observational` label contains:
- `Identification_Area` and `Observation_Area`: product-level metadata.
- One or more `File_Area_Observational`: each binds a data file (`File/file_name`) to one or more data objects.

The most common data object is `Array` / `Array_2D_Image`. It declares:
- `Element_Array/data_type`: element type and byte order.
- `Axis_Array` children: dimension names, lengths, and sequence order.

Sprint 3 already parses the XML document and validates the PDS4 namespace. This sprint walks that parsed document and builds the corresponding `NC_FILE_INFO_T` structures.

## Tasks

### 1. Map `Identification_Area` to global attributes

For each direct child of `Identification_Area` that contains text, create a global attribute on the root group:
- Name: the element local name.
- Type: `NC_CHAR` (string).
- Value: the element text content.

Examples: `logical_identifier`, `version_id`, `title`, `information_model_version`, `product_class`.

### 2. Map `Observation_Area` to global attributes

Flatten `Observation_Area` similarly. For simple text-valued descendants, create global attributes named by element local name. Keep the mapping deterministic: breadth-first or direct children only; document the chosen rule in the skill file if it changes.

### 3. Create a child group per `File_Area_Observational`

For each `File_Area_Observational`:
- Read `File/file_name`.
- Create a child group of the root group named from the file name (without directory). Use `nc4_grp_add` or equivalent.
- Store the data file path (resolved relative to the XML label directory) in group-level PDS4 state.

### 4. Create dimensions and variables for `Array` / `Array_2D_Image`

For each `Array` or `Array_2D_Image` inside a file area group:
- Sort `Axis_Array` by `sequence_number`.
- Create one netCDF dimension per axis: name from `axis_name`, length from `elements`.
- Create one netCDF variable in the group: name from `Array/name` if present, otherwise a default such as `data`.
- Set the variable `nc_type` from `Element_Array/data_type` using the PDS4-to-netCDF type table.
- Record the PDS4 byte order (MSB/LSB) in the variable's PDS4 state for Sprint 6.
- If the label declares `unit` on the array, add a `units` attribute.

### 5. Populate netCDF-4 internal metadata structures

Use the existing `NC_FILE_INFO_T` / `NC_GRP_INFO_T` / `NC_DIM_INFO_T` / `NC_VAR_INFO_T` infrastructure used by the HDF5 and Zarr backends. Because the PDS4 dispatch table reuses `NC4_inq*`, `HDF5_inq_dim`, etc., building these structures is sufficient to make metadata inquiries work.

### 6. Update metadata dispatch functions

Replace any no-op metadata functions wired in Sprint 3 with real implementations or confirm that the `NC4_*` / `HDF5_*` reused functions now return the populated metadata. In particular ensure:
- `nc_inq` returns root group info.
- `nc_inq_dims` / `nc_inq_dim` return created dimensions.
- `nc_inq_vars` / `nc_inq_var` return created variables.
- `nc_inq_attnum` / `nc_inq_attname` / `nc_get_att` return global attributes.

### 7. Extend `test/tst_pds4_udf.c`

Add assertions for the `test_image.xml` sample:
- `nc_inq` returns 1 group (the `test_image.img` file area).
- Inside that group there are two dimensions (`Line`, `Sample`) of length 4.
- There is one float variable.
- Global attributes include `title`, `logical_identifier`, etc.

### 8. Keep data reading disabled

`NC_PDS4_get_vara()` continues to return `NC_EINVAL`. This is expected and tested by absence of data-read tests.

### 9. Update documentation

- Remove or narrow the "no metadata reading" limitation in `docs/prd.md`.
- Update `.devin/skills/pds4/SKILL.md` with the exact array metadata mapping rules.
- Mention in `README.md` that PDS4 array metadata is now readable.

## Notes

- The current sample `test_image.xml` uses `Array_2D_Image` with `Last Index Fastest`. The dimension order is already C-order, so no reversal is needed beyond ordering by `sequence_number`.
- Only `Product_Observational` products are required to work; other PDS4 product classes remain out of scope.
- Global attribute flattening should be conservative: preserve simple string values; do not attempt to serialize complex nested elements unless they are unambiguously text-valued.
