# NEP v1.5.6 - .ncrc Fixes, CDF Magic Number Documentation, and Test Improvements

> Fixes `.ncrc` template configuration for separate plugin libraries, refines conditional `.ncrc` installation, improves CDF UDF autoloading tests, and adds detailed documentation on CDF magic number handling limitations in netcdf-c.

---

## Highlights

- **`.ncrc` plugin library fix** (#141) - Corrected `.ncrc.in` template to reference `libncgeotiff.so` and `libnccdf.so` individually instead of the monolithic `libnep`
- **`.ncrc` rename and conditional install** (#140) - `nep.ncrc` renamed to `.ncrc`; installation now conditional on `HAVE_GEOTIFF` or `HAVE_CDF` being enabled
- **Format-specific `.ncrc` templates** (#142) - New `.ncrc-geotiff.in` and `.ncrc-cdf.in` split out for per-format UDF configuration; main `.ncrc.in` reduced to header comment only
- **CDF magic number documentation** (#142) - New `docs/netcdf-c-cdf-magic-improvements.md` details the two bugs in netcdf-c 4.10.0 that prevent CDF files from autoloading via `nc_open()` without an explicit `NC_UDF2` flag
- **Test fixes** (#142) - `tst_cdf_udf.c` and `tst_imap_mag.c` updated; `tst_cdf_ncrc_autoload.c` cleaned up

## Bug Fixes

### .ncrc Template References Wrong Libraries (#141)

The `.ncrc.in` template generated after v1.5.5 incorrectly listed the monolithic `libnep` as the UDF plugin library for all format entries. Fixed to use the correct per-format shared libraries:

- `NETCDF.UDF0` / `NETCDF.UDF1` (GeoTIFF) → `libncgeotiff.so`
- `NETCDF.UDF2` (CDF) → `libnccdf.so`

Also added `.ncrc` (the generated output) and the autoload test binaries to `.gitignore`.

### Test File Corrections (#142)

- `tst_cdf_udf.c` — fixed UDF registration call arguments
- `tst_imap_mag.c` — fixed UDF registration call arguments
- `tst_cdf_ncrc_autoload.c` — removed stale include

## .ncrc Enhancements

### Rename: nep.ncrc → .ncrc and Conditional Installation (#140)

- `nep.ncrc.in` renamed to `.ncrc.in` throughout the build systems
- CMake: `.ncrc` is only installed when `-DHAVE_GEOTIFF=ON` or `-DHAVE_CDF=ON`
- Autotools: new `BUILD_NCRC` conditional; installation skipped when neither format is enabled
- `--with-ncrc-dir` configure option and `-DNEP_NCRC_INSTALL_DIR` CMake variable still supported
- Documentation in `README.md` and `docs/build-options.md` updated to use `.ncrc` filename
- `PKG_CONFIG_PATH` auto-discovery added to `configure.ac`

### Format-Specific Templates (#142)

Two new templates break out per-format UDF configuration:

- **`.ncrc-geotiff.in`** — UDF0 (BigTIFF `II+`) and UDF1 (standard TIFF `II*`) entries pointing to `libncgeotiff.so`
- **`.ncrc-cdf.in`** — UDF2 (NASA CDF `\xCD\xF3`) entry pointing to `libnccdf.so`

The main `.ncrc.in` is now a minimal template containing only the reference documentation comment, avoiding accidental hard-coded library paths in the default installed file.

## Documentation

### CDF Magic Number Improvements for netcdf-c (#142)

New `docs/netcdf-c-cdf-magic-improvements.md` documents two bugs in netcdf-c 4.10.0 that affect CDF autoloading:

1. **`strlen()` truncates magic numbers containing null bytes** — The CDF magic number `\xCD\xF3\x00\x01` contains a null byte at index 2. `nc_def_user_format()` uses `strlen()` and `strncpy()`, silently storing only 2 bytes. The document recommends replacing these with `memcpy` using `NC_MAX_MAGIC_NUMBER_LEN`.

2. **`strncmp()` in format detection ignores trailing bytes** — The dispatch layer compares only `strlen(magic)` bytes, so a 2-byte truncated magic can match unrelated file types. The document recommends comparing the full `NC_MAX_MAGIC_NUMBER_LEN` bytes.

This document is intended to support an upstream netcdf-c bug report and patch contribution.
