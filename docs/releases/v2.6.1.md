# NEP v2.6.1 - No More Worlds to Conquer

> "Alexander wept — for there were no more worlds to conquer." Fixes a regression in UDF self-registration for FITS and PDS4 format handlers introduced in v2.6.0, restoring correct behavior for both direct-call and `.ncrc` autoload initialization paths.

---

## Highlights

- **FITS and PDS4 UDF handlers now correctly register via both code paths** (#292) — `NC_FITS_initialize()` and `NC_PDS4_initialize()` now call `nc_def_user_format()` for direct invocation *and* return the dispatch table pointer for `.ncrc` self-load, making both paths work simultaneously
- **Fortran FITS test updated for new return type** (#292) — `ftst_fits_udf.F90` interface block now uses `type(c_ptr)` and `c_associated()` instead of `integer(c_int)` comparison to `NC_EINVAL`
- **Header macros corrected** (#292) — `FITS_INIT_OK()` and `FITS_INIT_AND_ASSIGN()` in `include/fitsdispatch.h` updated to test for `NULL` instead of `NC_NOERR`

---

## Bug Fixes

- **`NC_FITS_initialize()` returned dispatch pointer but never registered the handler** (#292) — calling `nc_open()` after `NC_FITS_initialize()` returned `NC_ENOTNC` ("Unknown file format") because the format was not registered with NetCDF-C; fixed by restoring the `nc_def_user_format()` call while also returning the dispatch table pointer
- **`NC_PDS4_initialize()` same regression** (#292) — identical fix applied to `src/pds4dispatch.c` and `include/pds4dispatch.h`
- **Fortran interface ABI mismatch** (#292) — `ftst_fits_udf.F90` declared `NC_FITS_initialize` as returning `integer(c_int)` after the return type changed to `NC_Dispatch*`; fixed with `type(c_ptr)` binding

---

## API Changes

- `NC_FITS_initialize(void)` return type changed from `int` to `NC_Dispatch*` — callers should check `!= NULL` for failure rather than `!= NC_NOERR`
- `NC_PDS4_initialize(void)` return type changed from `int` to `NC_Dispatch*` — same pattern
- Both functions remain safe to call when the handler is already registered via `.ncrc`; the `NC_EINVAL` from a duplicate `nc_def_user_format()` call is silently ignored

## Known Limitations

- `nc_get_vars()` and `nc_get_varm()` rely on default dispatch fallbacks (no strided native reads)
- PDS4: `Table_Delimited` row-count derived from label `<records>` only

---

## Breaking Changes

Callers of `NC_FITS_initialize()` or `NC_PDS4_initialize()` that checked for `NC_NOERR`/`NC_EINVAL` integer return values must be updated to check the returned pointer for `NULL`. The `FITS_INIT_OK()` and `FITS_INIT_AND_ASSIGN()` macros in `include/fitsdispatch.h` handle this automatically.
