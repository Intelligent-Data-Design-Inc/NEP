# v1.5.5: Header File Cleanup

**Labels:** enhancement, v1.5.5, header-cleanup

## Overview

NEP currently copies 14 private header files from `netcdf-c/include` into its own `include/` directory. These headers expose internal netcdf-c data structures (`NC_FILE_INFO_T`, `NC_GRP_INFO_T`, `NCindex`, etc.) that the GeoTIFF, CDF, and GRIB2 UDF handlers use to build the in-memory netcdf-4 metadata model.

This version begins a systematic, incremental effort to reduce and eventually eliminate this dependency. The goal is not to complete the cleanup in one version, but to establish a clear map of what is needed and why, remove what can be removed safely now, and create an abstraction layer that isolates the remaining dependency.

## Why These Headers Are Needed

NEP's UDF handlers must populate the netcdf-4 in-memory metadata model (groups, variables, dimensions, attributes) so that the standard NetCDF API can query them. This requires access to internal structures defined in `nc4internal.h` and related headers. There is currently no public API in netcdf-c to build these structures from outside the library.

The long-term solution requires either:
1. A public API in netcdf-c for UDF handlers to register metadata (requires upstream changes)
2. A thin NEP abstraction layer that wraps the internal struct access, isolating the dependency

## Sprints

- **Sprint 1**: Audit, dependency map document, version-stamp all copied headers, eliminate any dead/unused includes
- **Sprint 2**: Create NEP abstraction layer (`include/nep_nc4.h`) wrapping internal struct access; explore minimal upstream netcdf-c changes to support a public metadata-building API
