title: NetCDF-4 Read Throughput: Chunk Shape vs. Access Pattern

caption: Results from the chunking_performance.c example program in the NetCDF Expansion Pack. Dataset is a 100x36x72 NC_FLOAT temperature array (approximately 10 MB). NetCDF-4 stores chunked data as fixed-size tiles on disk; every read that crosses a chunk boundary forces the entire chunk to load. Time-optimized chunks (1x36x72) store one complete horizontal field per chunk: time-slab access loads exactly one chunk per time step (fast), but extracting a column profile requires loading all 100 chunks (slow). Column-optimized chunks (100x1x1) store the full time series for one grid point per chunk: column-profile access loads exactly one chunk per point series (fast), but assembling a horizontal field requires loading all 2,592 chunks (slow). Neither chunk shape is universally best; the key principle is to match chunk layout to the dominant access pattern. Higher throughput is better.

alt_text: Grouped bar chart with two clusters on the x-axis representing two access patterns: time slab (100 horizontal fields) and column profile (2,592 point series). Each cluster contains two bars, one solid white and one with diagonal hatching, representing time-optimized chunks (1x36x72) and column-optimized chunks (100x1x1) respectively. The y-axis shows read throughput in MB/s on a logarithmic scale. The solid bar is tall in the time-slab cluster and short in the column-profile cluster; the hatched bar is short in the time-slab cluster and tall in the column-profile cluster, illustrating that chunk shape must match the access pattern for good performance.
